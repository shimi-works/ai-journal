Case Study — PWA / LLM

AI Journal

書くことはシンプルに、振り返ることはAIに。
3分で書いた日記に要約・タグ・感情が付き、週・月・年の振り返りが自動で書き上がる。

制作 / 個人 形態 / 単一HTML・PWA・ノーサーバー LLM / Google Gemini（無料枠） 保存 / localStorage 2026

これは何か

Overview

日記を書く画面にはタイトル・本文・保存と、タップだけの点数・気分しかない。 保存した瞬間に裏でAIが動き、要約・タグ・感情推定と天気が自動で付く。 貯まった日記からは週・月・年の振り返りが「今週は〜な週でした」という文章で生成され、 1年前の同じ日の日記が「今日のこの日」として顔を出す。 「書くのは3分、振り返りは自動」を一つのPWAで完結させた。

目的は日記を書くことではなく、過去の自分を振り返りやすくすること。 整理・分析・要約という「続かない原因」をすべてAIに移し、人間には書くことだけを残した。

設計思想

Design Philosophy

入力の少なさを機能より優先した。 日記アプリは機能を足すほど書くのが億劫になる。だから「シンプルさを絶対に崩さない」を最上位の要件に置き、 入力はタイトル（任意）・本文・2タップの点数と気分だけ。タグ付け・要約・感情記録・まとめ書きといった 手間になる作業は全部AIの仕事にした。手動入力を増やしたのは点数と気分だけで、 それも「AI推定より本人の申告が正しい」ものに限っている。

AIなしでも日記アプリとして成立させた。 AI処理も天気取得も保存の後追いで走り、失敗しても日記は無傷。APIキーがなくても 書く・読む・検索する・エクスポートするは全部動く。ローカルファースト（データは端末の中、 ネットがなくても書ける）を守ることで、AIは「なければ困る依存」ではなく「あれば嬉しい増幅」になっている。

振り返りを"体験"として設計した。 書き溜めた日記は、読み返されなければただのログになる。週・月・年のまとめ、気分と点数のグラフ、 書いた日が一目でわかるカレンダー、1年前の今日——どれも「開いたら過去の自分に出会ってしまう」 きっかけとして一貫させた。語彙ラボで「学習の継続」を設計したのと同じ手つきで、ここでは「記録と振り返りの循環」を設計している。

技術構成

Architecture

配信形態
単一HTMLファイル＋Service WorkerのPWA。ビルド工程・フレームワーク・依存パッケージなし。ホーム画面に追加すればオフラインでも起動する。
永続化
localStorage にバージョン付きスキーマで全データ。v1→v2→v3のチェーンマイグレーションで旧バックアップも読める。JSON/Markdown書き出し・Web Share APIでiCloud等へ共有・他アプリからのテキスト移行インポート。
AI 呼び出し
ブラウザから Gemini API を直接叩く（BYOK）。構造化出力（responseSchema）でJSONを安定生成し、無料枠の上限(429)では別モデルへ自動巡回。語彙ラボで確立した構成の再利用。
外部データ
天気は Open-Meteo（キー不要・無料）。保存時に非同期で取得し、当日の日記にだけ付与する。
グラフ描画
外部ライブラリなしのSVG手組み。点数の折れ線と気分のドットは、ダーク面上で色覚多様性の分離を検証した配色。

主要な技術的判断

Key Decisions — 課題 / 判断 / 理由

AIを「裏方」に徹底して置く

課題
AI処理を保存と直列にすると、待ち時間と失敗がそのまま「書く体験」を壊す。無料枠のAPIは失敗が日常でもある。
判断
保存は即時にローカルへ確定し、要約・タグ・感情推定と天気は保存後に非同期で後追い付与。失敗したら黙って諦め（日記は無傷）、日記画面からいつでも手動でやり直せる。
理由
日記アプリの生命線は「書くのが一瞬で終わること」。AIの都合をユーザーに1秒も払わせない構造にした。

手動入力とAI推定の優先順位

課題
気分はAIで推定できるが、本人の実感と食い違うことがある。かといって毎日必ず入力させると続かない。
判断
気分は手動4択（2タップ・任意）が正、AI推定は未入力の日だけの補完という優先順位に固定。グラフ・カレンダー・統計は全部この合成値を使い、AI由来には「AI推定」と明記する。
理由
「入力を増やさない」と「データの正しさ」は衝突する。優先順位を1本のルールに落とすことで、両方をほぼ満たした。

無料枠に耐える、まとめ生成の階層設計

課題
週・月・年のまとめを本文全文から作ると、日記が増えるほどトークンが膨らみ、年間まとめは無料枠では成立しなくなる。
判断
まとめの素材を各日のAI要約にし、さらに年間まとめは月間まとめから生成する階層構造にした。1日→1要約、1ヶ月→1まとめ、と各段が定数サイズに畳まれていく。
理由
日記が10年分になっても、年間まとめの入力は「12個の月間まとめ」のまま。書き続けるほど破綻する設計を最初から避けた。

iOS実機で判明した、日付選択の作り直し

課題
過去日にさかのぼって書く機能が、iPhoneでは「1日ずつ戻るしかない」状態だった。iOS Safariは input[type=date] の showPicker() に非対応で、隠したinputをクリックさせる迂回も効かない。
判断
ネイティブピッカーへの依存を捨て、月送り・未来日無効・外側タップで閉じる自前のカレンダーポップアップに全面変更した。
理由
ヘッドレス検証では通っても実機で使えなければ意味がない。実機フィードバックが出た日のうちに、迂回ではなく根本から作り直した。

「消えない」ことを機能として設計する

課題
日記は消えたら取り返しがつかない。しかしlocalStorageはブラウザ都合で消え得るし、バックアップは面倒だと実行されない。
判断
多層で守る——①ブラウザに永続化を要求（storage.persist）②最終バックアップ日時と未保存件数を常時記録③7日放置すると未保存件数つきのリマインダー④Web Share APIで共有シート→iCloud Driveへ1タップ保存⑤Markdownでも書き出せる将来も読める形式。
理由
「バックアップしてください」と書くのは機能ではない。忘れた頃に、1タップで済む形で差し出すところまでが設計。

ヘッドレスブラウザによる回帰テストの自動化

課題
1日に何度も機能追加する開発ペースで、手動確認だけでは「動いていたものが壊れる」を防げない。実際に過去、編集ミスで起動不能のままpushした事故がある（別作品）。
判断
ヘッドレスブラウザで実クリックをdispatchするインタラクションテスト（50項目超）を毎回実行。保存・検索・ロック・マイグレーション・AI処理（APIはスタブ）・バックアップまで、ユーザー操作の粒度で回帰を検証してからpushする。
理由
個人開発こそ「自分しか気づけない」。テストをアプリ本体と同じ速度で育てることで、この開発ペースが維持できている。

制約と割り切り

Trade-offs

承知の上のトレードオフ

  • 書くリマインド通知は載せていない: 通知で書かせる日記は続かない、が設計判断。代わりに「開いたら1年前の今日に出会う」引力で開く理由を作っている。
  • APIキーは端末内保存＝個人利用が前提: 不特定多数に配るならプロキシが要る。個人の日記帳という用途には、各自の鍵で無料のまま完結する方が合う。
  • パスワードは覗き見防止の簡易ロック: データ自体の暗号化ではないとUIに明記し、忘れたときは日記を守る方向（データ保持でロック解除）に倒した。
  • 週・月・年まとめは手動生成: サーバーがないので「日曜の夜に自動で届く」はできない。開いたときに1タップで生成する形で代替（自動化は次の課題）。

この作品の位置づけ

Context

「そのままでは続かないものを、続けたくなる体験に変える」

わたしの制作物に共通する軸は、学習と創作の体験を設計すること。 語彙ラボが「学習の継続」を、Chord Labが「創作の入口」を設計したのに対し、 AI Journalは「記録と振り返りの循環」を設計した3本目にあたる。 書く負荷を限界まで下げ、読み返す理由をAIと仕掛けで作る—— 日記が続かない原因を一つずつ体験の設計で潰していった作品。
